1. Executive Summary


This document defines the strategy, scope, phased plan, resource requirements, risk register, and governance framework for delivering a TSANet Connect 3.0 integration with the Zendesk Support platform. The integration will enable TSANet member companies that run Zendesk as their primary ITSM or CRM to participate in multi-vendor support collaboration directly within their existing tooling, without context-switching to the TSANet Web App.
The project is structured across three phases spanning 2026 to 2027, progressing from a documented community integration pattern to a full TSANet-supported connector distributed via the Zendesk Marketplace. This mirrors the maturity path taken by the Salesforce and Microsoft Dynamics connectors, and directly advances several Connect 3.0 roadmap objectives including Strategic Integrations, Service Provider expansion, and Developer Enablement.

2. Strategic Context & Business Case


2.1  Alignment with TSANet Connect 3.0 Roadmap
The TSANet 2026 Plans document identifies Strategic Integrations as a distinct integration tier above Community Integrations, sitting alongside the core TSANet-supported connectors for Salesforce, Microsoft Dynamics, and ServiceNow. The Connect 3.0 initiative has four explicit goals that a Zendesk connector directly advances:
Connect 3.0 Goal
How Zendesk Connector Addresses It
Integrated Members
Zendesk members gain native collaboration without swivel-chairing to the Web App
Service Provider Expansion
MSPs and TPMs commonly run Zendesk; this connector is the infrastructure for Service Provider use cases launching 2H2026
Developer Enablement
Reference implementation and sample code fulfills the roadmap goal of documentation and sample code for developers
Certified Integration Support
Phase 2 Marketplace app becomes the first community-to-certified integration pathway, establishing the certification model

2.2  Market Rationale
Zendesk is the third most widely deployed customer support platform globally, with significant penetration among technology vendors — precisely the TSANet target member profile. Key market dynamics supporting this investment:
	•	Zendesk holds strong market share among mid-market and enterprise technology companies not on Salesforce or Dynamics
	•	MSPs and TPMs — the target audience for TSANet's Service Provider expansion — frequently standardize on Zendesk for its multi-client ticket management
	•	The absence of a Zendesk connector is a friction point in TSANet member recruitment conversations with Zendesk-based companies
	•	The Zendesk Marketplace provides a zero-cost distribution channel with built-in discoverability for 160,000+ Zendesk customer accounts


Strategic Positioning

A Zendesk connector positions TSANet as the default collaboration layer for MSPs entering the Service Provider program, giving the 2H2026 expansion concrete integration infrastructure from day one.

3. Project Objectives & Success Criteria


3.1  Project Objectives
	•	Phase 1: Phase 1: Publish a production-ready Zendesk integration reference guide, an exportable ZIS flow template (inbound path, no code required), and a single-file Cloudflare Worker for outbound case submission — enabling any TSANet member to self-integrate within 2 weeks of reading the documentation
	•	Phase 2: Deliver a distributable Zendesk App Framework (ZAF) sidebar app and companion automation package, listed on the Zendesk Marketplace, supporting the full bi-directional TSANet collaboration lifecycle
	•	Phase 3: Elevate the connector to a TSANet-supported integration — owned, maintained, and iterated by TSANet — equivalent to the Salesforce and Dynamics connector tier
	•	Cross-cutting: Establish the community-to-certified integration pathway as a repeatable model for future platform connectors

4. Scope Definition


4.1  In Scope
	•	Documentation: Zendesk-specific integration reference guide published to TSANet GitBook
	•	ZIS flow template: exportable JSON config for the full inbound event path (status changes, notes, INFORMATION alerts, SLA breach) — importable by any Zendesk admin with no code required
	•	ZAF Sidebar App: embedded ticket-page panel showing TSANet collaboration status, partner details, SLA timer, and action buttons
	•	Zendesk Trigger & Automation package: pre-built Zendesk configuration for outbound webhook firing
	•	ZIS Inbound Webhook integration: receive TSANet Connect 3.0 webhook events and write back to Zendesk tickets
	•	Full bi-directional collaboration lifecycle: case creation, acceptance, information request/response, notes, SLA alerts, closure
	•	Token storage: TSANet collaboration token stored as Zendesk custom ticket field
	•	Note mapping: TSANet notes → Zendesk comments (internal vs. public mapped to Partner Only vs. Customer+Partner visibility)
	•	Status/SLA reflection: TSANet case status and SLA deadline surfaced in Zendesk ticket fields and sidebar UI
	•	Zendesk Marketplace listing and review process
	•	Beta program: minimum 3 pilot member validations before general availability
	•	Phase 3: TSANet-owned GitHub repository, release management, and connector maintenance process

4.2  Out of Scope
	•	Zendesk Sell (CRM) integration — Zendesk Support only in Phases 1 and 2
	•	Zendesk Guide or Help Center integrations
	•	Auto-approval flows based on Zendesk Organization entitlement lookup (deferred to Phase 3)
	•	Multi-language localization of the ZAF app UI
	•	Custom branding / white-labeling of the connector for individual TSANet members
	•	Integration with Zendesk Sunshine / custom objects (evaluated in Phase 3 roadmap planning)
	•	Changes to the TSANet Connect REST API (scope is integration to existing API endpoints)

7. Work Breakdown Structure


Phase 1: Community Integration
Q1 – Q2 2026  |  Owner: TSANet Development  |  Effort: Low

1.1  Documentation Package
	•	Author the Zendesk Integration Reference Guide for TSANet GitBook
	•	Document the three Zendesk integration surfaces: outbound webhooks, ZIS inbound webhooks, and ZAF sidebar app
	•	Define API field mapping: Zendesk ticket fields ↔ TSANet collaboration request model
	•	Write the Email-to-Case bridge guide for Zendesk members not yet ready for API integration
	•	Publish Postman collection extension with Zendesk-specific request examples

1.2  ZIS Flow Template & Cloudflare Worker
	•	Create the ZIS integration container and register the inbound webhook endpoint (the URL TSANet will POST all collaboration events to)
	•	Author and upload the ZIS flow JSON template handling all four TSANet event types: status.changed, note.created, information.requested, and sla.breach
	•	Publish the ZIS flow template as an exportable JSON config to TSANet GitHub so any admin can import and configure with their own TSANet credentials — no code required
	•	Build the Cloudflare Worker (single JS file, ~150 lines): chains TSANet API calls for the outbound path — login, form fetch, case submission — and writes the returned token back to the Zendesk ticket custom field
	•	Implement Zendesk webhook signature verification (HMAC-SHA256) and graceful error handling in the Worker — failed submissions write an actionable error comment back to the ticket
	•	Publish the Worker to TSANet GitHub with wrangler.toml, .env.example, and a 10-minute deployment guide — deployable to Cloudflare Workers free tier with zero ongoing hosting cost

1.4  Pilot Member Preparation
	•	Account managers identify 3+ Zendesk-based member companies willing to self-integrate using the guide
	•	TSANet issues Beta environment credentials to pilot members
	•	Collect structured feedback on documentation gaps and integration friction points

Phase 2: Strategic Integration — ZAF App & Marketplace
Q2 – Q4 2026  |  Owner: TSANet Dev + Partner  |  Effort: Medium

2.1  Technical Architecture & Design
	•	Finalize ZAF app architecture: sidebar app, companion trigger package, ZIS webhook listener
	•	Design token storage schema: TSANet token as Zendesk custom ticket field (tsanet_token)
	•	Design status reflection: TSANet case status mapped to Zendesk ticket tags and custom field
	•	Design note visibility mapping: Partner Only → internal comment, Customer+Partner → public reply
	•	Design one-to-many model: single Zendesk ticket linked to multiple TSANet collaboration cases
	•	Define SLA surfacing: TSANet respondBy deadline visible in sidebar as countdown

2.2  ZAF Sidebar App Development
	•	Build app scaffold using Zendesk Apps Framework SDK
	•	Implement app authentication: API token entry in Zendesk app settings, mapped to TSANet API user
	•	Build TSANet case list panel: shows all active collaborations linked to the current ticket
	•	Build action buttons: Submit Collaboration, Accept, Request Info, Provide Info Response, Close
	•	Build note feed panel: shows TSANet notes with visibility indicator, priority, and timestamp
	•	Build SLA indicator: color-coded timer showing time remaining to initial response deadline
	•	Build INFORMATION status prompt: surfaces clearly when partner has requested additional information

2.3  Zendesk Automation Package
	•	Create Zendesk Trigger: fires outbound webhook when agent tags ticket with tsanet_initiate
	•	Create Zendesk Automation: fires outbound webhook when TSANet-related ticket SLA nears breach
	•	Register ZIS Inbound Webhook endpoint to receive TSANet Connect 3.0 webhook events
	•	Build ZIS workflow: parse TSANet webhook event → call Zendesk Tickets API to update ticket

2.4  Beta Program
	•	Enroll minimum 3 pilot member organizations (target: Standard, Premium, and Elite tier representative)
	•	Conduct end-to-end test: outbound case creation from Zendesk ticket through TSANet to partner and back
	•	Validate INFORMATION status handling, note bidirectionality, SLA timer accuracy, and closure flow
	•	Collect structured feedback using a test case tracking sheet; triage and resolve P1 issues before GA

2.5  Zendesk Marketplace Submission
	•	Register as a Zendesk Technology Partner
	•	Prepare Marketplace listing: description, screenshots, privacy policy, support contact
	•	Submit app for Zendesk security and functionality review (estimate 4–8 weeks)
	•	Address review feedback and complete any required remediation
	•	Publish to Zendesk Marketplace; announce to TSANet member community via newsletter and GitBook

Phase 3: TSANet Supported Connector
Q1 – Q3 2027  |  Owner: TSANet Engineering  |  Effort: Medium–High

3.1  Connector Ownership & Repository
	•	Establish connector repository at github.com/tsanetgit/zendesk (parallel to /SFDC repo)
	•	Define release management process: semantic versioning, changelog, release notes
	•	Migrate Phase 2 codebase into repo with full attribution
	•	Draft contributor guidelines for community pull requests

3.2  Feature Expansion
	•	Implement auto-approval flows: entitlement check via Zendesk Organization or custom object lookup
	•	Support multiple simultaneous TSANet collaborations per Zendesk ticket in the UI
	•	Add file attachment support: integrate Zendesk file attachments with TSANet HTTPS attachment endpoint
	•	Add scheduled reconciliation poll (hourly) as webhook delivery safety net

3.3  Certification & Governance
	•	Submit connector for review by Connect 3.0 Technology Committee
	•	Achieve TSANet Supported Connector status, listed alongside Salesforce, Dynamics, ServiceNow in documentation
	•	Add Zendesk to TSANet GitBook connector selection decision matrix
	•	Establish ongoing maintenance cadence: quarterly release cycle aligned with TSANet platform updates

10. Key Technical Design Decisions


The following design decisions are architectural commitments that will shape the entire integration. They should be reviewed and formally approved before Phase 2 development begins.
#
Decision
Rationale
Implication
D1
Store TSANet token as a Zendesk custom ticket field (tsanet_token)
Token is the API primary key for all TSANet operations — must be persisted on the Zendesk ticket to enable any subsequent API call
Custom ticket field must be created during app installation; field ID stored in app settings
D2
Map TSANet case status to Zendesk ticket tags (e.g. tsanet_accepted, tsanet_closed)
Tags are the standard Zendesk mechanism for driving trigger/automation behavior; enables agents to build views based on TSANet state
Tag management logic must handle both add and remove operations on status transitions
D3
Map note visibility: Partner Only → Zendesk internal comment; Customer+Partner → public reply
Zendesk's internal/public comment distinction maps directly to TSANet note visibility levels; preserves confidentiality intent
UI must present a visibility selector when composing notes; default to internal (Partner Only)
D4
Use hybrid architecture: TSANet webhooks for real-time + hourly reconciliation poll as safety net
Webhooks are the preferred Connect 3.0 pattern, but delivery failures are possible; reconciliation poll ensures no events are permanently missed
Integration requires both webhook endpoint registration and a scheduled polling service
D5
Model one Zendesk ticket to many TSANet collaborations (one-to-many)
A single customer issue may involve simultaneous escalations to multiple vendor partners; flattening to one-to-one loses critical context
Data model must support multiple tsanet_token values; sidebar app must display a list of collaborations
D6
Middleware is stateless; all state persisted in Zendesk ticket fields and TSANet API
Stateless Worker is horizontally scalable, easier to maintain, and avoids a separate database dependency
All lookups for token, status, and partner info go through Zendesk or TSANet API calls — no local cache
D7
INFORMATION status surfaced as a distinct UI alert in the sidebar app, not just a status field
INFORMATION is the most commonly missed status in integrations; agents must be prompted to respond, not just notified
Sidebar app must check for INFORMATION status on load and show an action banner with the Provide Info Response button

11.3  Change Control Process
Any change to scope, timeline, or resources follows this process:
	•	Change requestor documents the proposed change: description, rationale, impact on scope / timeline / resources
	•	Product Manager assesses impact and assigns priority (P1 = critical path, P2 = important, P3 = nice-to-have)
	•	If impact is minor (less than 3 days effort, no timeline shift): Product Manager approves and updates project log
	•	If impact is significant (more than 3 days or timeline shift): stakeholder review required before approval
	•	All approved changes are documented in the project change log with date, decision, and authority
